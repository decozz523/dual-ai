# Оценка проекта dual-ai

## Краткий вердикт
Проект уже выглядит как **рабочий MVP** с богатым UX, двухперсонным чатом и базовой monetization-логикой (free/plus/pro). По зрелости: **6.5/10** для MVP и **4.5/10** для production-ready.

## Что сделано хорошо
- Есть полноценный одностраничный интерфейс с домашним экраном, настройками, историей диалогов и мультиязычностью RU/EN.
- Продумана продуктовая логика: режимы диалога, шаблоны, экспорт, идеи, планы подписки и активация кода.
- Вынесен серверный прокси для OpenRouter (`api/chat.js`), чтобы не светить API-ключ в клиенте.
- Есть задел под авторизацию и хранение данных через Supabase.

## Основные риски
1. **Безопасность секретов**
   - В `bot.py` токены/ключи заданы как значения в коде-заглушке; при неправильной эксплуатации это быстро приводит к утечкам.
   - Нужна обязательная загрузка секретов через переменные окружения + валидация на старте.

2. **Сложность поддержки фронтенда**
   - `app.js` (очень большой монолит) сложно тестировать и безопасно менять.
   - Нет разделения на модули (state, api, ui, i18n, billing), высокий риск регрессий.

3. **Недостаток инженерных guardrails**
   - Практически пустой `README.md` — нет инструкций запуска, env-переменных, деплоя и troubleshooting.
   - Не видны линтеры/форматтеры/автотесты, из-за чего качество и скорость разработки будут падать при росте проекта.

4. **Устойчивость backend-слоя**
   - В `api/chat.js` нет явной обработки сетевых ошибок/таймаутов/ограничений, что может дать нестабильный UX при проблемах с внешним API.

## Приоритетный план улучшений (2–3 спринта)
1. **Sprint 1 (обязательно):**
   - Секреты только через `.env` (без значений в репо).
   - Расширить `README.md`: запуск, env, архитектура, команды.
   - Добавить ESLint + Prettier + базовый CI.

2. **Sprint 2:**
   - Разбить `app.js` на модули: `state/`, `api/`, `ui/`, `i18n/`, `billing/`.
   - Ввести централизованный обработчик ошибок и retry/backoff для вызовов API.

3. **Sprint 3:**
   - Добавить минимальные e2e/интеграционные тесты для ключевых сценариев (вход, чат, сохранение, апгрейд).
   - Улучшить наблюдаемость: structured-логи и простые продуктовые метрики.

## Итог
У проекта хороший продуктовый потенциал и заметно продуманный UI/UX. Следующий шаг — закрыть базовые инженерные риски (секреты, модульность, документация, тесты), чтобы перейти от «красивого MVP» к устойчивому продукту.
